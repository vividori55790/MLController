## 1. 코드 상단: 버전 관리 및 변경 이력 (Changelog)

파일의 가장 첫머리에 들어갈 양식입니다. AI에게 이 파일이 '흐름과 역사'가 있는 파일임을 인지시키고, 함부로 과거 유산을 지우지 못하게 압박하는 역할을 합니다.

```text
======================================================================
[FILE METADATA & VERSION TRACKING]
- Current Version: v1.2.0 (2026-05-21)
- Target Environment: Production / Node.js v20.x
- Integrity Check: DO NOT delete any existing functions unless explicitly requested.
======================================================================
[CHANGELOG - NEVER DELETE THIS HISTORY]
* v1.2.0 (2026-05-21) - User: [변경 사항 요약 작성]
  - Added: 새로운 기능 명세
  - Fixed: 버그 수정 내용
  - CRITICAL KEEP: 이 버전에서 절대 유지해야 하는 기존 기능 명시
* v1.1.0 (2025-11-14) - User: Initial stable release.
======================================================================

```

// - Edge Cases Handled: 타임아웃 3회 초과 시 로컬 스토리지 우회 백업 (체크 필수)
// ==================================================================
function criticalDataSync() {

## 3. 고통 방지를 위한 추가 조치: AI 행동 지침 (AI Instructions)

요청하신 두 가지 외에, **AI의 '자기 마음대로 예측하기' 성향을 통제하기 위해 파일 최상단이나 프롬프트에 반드시 포함해야 할 추가 규칙**입니다. 이 규칙들이 있어야 AI가 디테일을 뭉개지 않고 꼼꼼하게 점검합니다.

```text
======================================================================
[AI INSTRUCTIONS - RULES OF ENGAGEMENT]
AI는 이 코드를 수정하거나 이어서 작성할 때 다음 규칙을 반드시 준수해야 합니다.

1. 전수 검사 원칙 (No Laziness):
   - "이전과 동일", "// ... 기존 코드 생략" 같은 표현은 절대 금지합니다.
   - 사소한 변경이라도 파일 전체를 생략 없이 온전한 코드로 출력하세요. 생략하는 과정에서 디테일이 유실됩니다.

2. 보존 우선 원칙 (Preservation First):
   - 현재 구현하려는 기능과 직접적인 관련이 없어 보이는 함수나 변수라도, 삭제 요청이 없다면 그대로 유지해야 합니다.
   - 이해가 가지 않는 코드가 있다면 삭제하지 말고 주석으로 질문하세요.

3. 영향도 사전 평가 (Impact Analysis):
   - 코드를 수정하기 전, 본인이 수정할 내용이 기존의 다른 기능(특히 상단 Changelog에 언급된 CRITICAL KEEP 기능들)에 어떤 영향을 미칠지 스스로 예상하고, 그 검토 결과를 코드 출력 전에 1줄로 먼저 설명하세요.

4. 예외 케이스 예측 (Edge-Case Thinking):
   - 해피 패스(정상적인 상황)만 고려하지 마세요. 데이터가 null일 때, 네트워크가 끊겼을 때, 사용자가 더블 클릭했을 때 등 발생할 수 있는 오류를 직접 점검하고 방어 코드를 짜세요.

5. 빌드 및 패키징 제약 조건 (Build & Packaging Restrictions):
   - 사용자가 대화창에서 명시적으로 ".exe로 빌드해줘" 혹은 "PyInstaller 빌드(또는 컴파일)를 수행해줘" 라고 승인하여 명령하지 않은 경우, 절대로 자체적인 판단 하에 빌드 명령(PyInstaller, build.ps1 스크립트 실행 등)을 수행하거나 제안하지 마세요. 불필요한 빌드 시간 낭비와 시스템 리소스 소모를 방지하기 위한 절대 제약사항입니다.

6. 런타임 디버깅 및 실행 검증 의무 (Runtime Debugging Loop):
   - 작성한 파이썬 코드를 눈으로만 검사하지 마. 반드시 터미널에서 python3 main.py를 직접 실행하고(Run), 발생하는 Traceback 및 에러 로그 전체를 수집해. 만약 실행 오류가 나면 오류 메시지를 바탕으로 코드를 수정하고, 오류 없이 100% 정상 작동할 때까지 이 실행-수정 루프(Runtime Debugging Loop)를 자율적으로 반복해줘.
======================================================================
